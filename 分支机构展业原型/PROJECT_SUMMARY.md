## 🔑 核心特性

### 1. 谁维护谁查看原则 ⭐⭐⭐⭐⭐

**实现方式：**
```javascript
// 查询时自动过滤
filtered = filtered.filter(c => c.branch === currentUserBranch);

// 编辑时权限检查
if (company.branch !== currentUserBranch) {
    showToast('权限不足', 'error');
}

// 删除时权限检查  
if (company.branch !== currentUserBranch) {
    showToast('权限不足', 'error');
}
```

**体现的价值：**
- ✅ 数据安全 - 防止跨分支操作
- ✅ 职责清晰 - 明确归属
- ✅ 可审计 - 记录维护人员
- ✅ 易管理 - 分权分域

### 2. 企业库关联机制 ⭐⭐⭐⭐

**关联流程：**
```
维护企业库 → 创建展业记录 → 下拉选择企业 → 自动填充信息 → 保存记录
```

**技术实现：**
- 企业库独立管理（companies数组）
- 展业记录引用企业名称
- 下拉列表动态生成
- 权限过滤应用到下拉列表

**核心价值：**
- ✅ 数据重用 - 避免重复输入
- ✅ 数据一致性 - 同一企业信息统一
- ✅ 快速操作 - 展业时直接选择
- ✅ 易于分析 - 同企业历史查询

### 3. 自动留痕机制 ⭐⭐⭐⭐⭐

**留痕内容：**
```
创建时：
  ├─ 时间戳
  ├─ 操作人
  ├─ 操作类型（新增）
  └─ 操作摘要

修改时：
  ├─ 变更的字段
  ├─ 修改前的值
  ├─ 修改后的值
  ├─ 操作时间
  └─ 操作人
```

**代码示例：**
```javascript
record.changes.push({
    type: '修改',
    time: new Date().toLocaleString('zh-CN'),
    operator: personName,
    content: `字段1: 旧值 → 新值; 字段2: 旧值 → 新值`
});
```

**应用场景：**
- ✅ 审计追踪 - 完整操作记录
- ✅ 版本恢复 - 看到所有变更
- ✅ 责任认定 - 知道谁改了什么
- ✅ 问题溯源 - 追踪数据变化

### 4. 完整的权限架构 ⭐⭐⭐⭐

**当前实现：**
```
当前用户 → 北京分部
  ├─ 可维护 → 北京分部企业
  ├─ 可查看 → 北京分部企业
  ├─ 可新增 → 北京分部展业
  └─ 可编辑 → 自己创建的展业
```

**扩展点：**

## 🔍 使用场景和演示流程

### 场景1：从零开始建立展业体系

**步骤：**
1. 打开系统
2. 进入"企业信息"页面
3. 点击"➕ 新增企业"
4. 填写企业的完整信息（包括财务数据）
5. 保存企业
6. 进入"新增展业记录"
7. 从下拉列表选择企业
8. 填写展业信息
9. 上传留痕材料
10. 保存展业记录

**期望结果：**
- ✅ 企业库中新增1条企业
- ✅ 展业记录中新增1条记录
- ✅ 两者自动关联

### 场景2：展业进展追踪

**步骤：**
1. 进入"展业查询"
2. 找到某条展业记录
3. 点击"编辑"
4. 更新推进状态和进展说明
5. 上传新的留痕材料
6. 保存记录
7. 点击"查看"查看变更历史

**期望结果：**
- ✅ 变更历史自动记录所有修改

1. 进入"企业信息"

**期望结果：**
- ✅ 企业信息保持最新
- ✅ 历史展业记录可查看当时的企业信息

## 📞 常见问题速查

### Q: 企业库和展业记录的关系是什么？
A: 企业库是"主数据"，展业记录"引用"企业库。一个企业可以有多条展业记录，但每个展业记录只关联一个企业。

### Q: 如何切换不同分支用户？
A: 修改代码第7行的 `currentUserBranch` 变量即可。

### Q: 数据会不会丢失？
A: 不会。数据保存在浏览器的localStorage中，即使关闭浏览器也能恢复。

### Q: 能否与其他分支共享数据？
A: 当前版本不能。这是为了遵循"谁维护谁查看"的原则。

### Q: 能否修改已有企业的维护分支？
A: 不能。这是为了维护权限的一致性。如需修改，应删除后重新新增。
